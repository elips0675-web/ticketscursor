# Промты для разработки фич

> Используй как шаблоны при генерации/доработке кода через ИИ (Claude, GPT-4, Cursor и т.д.)

## Базовый промт-шаблон (для ЛЮБОЙ фичи)

```
Ты senior fullstack разработчик. Твоя задача — реализовать функционал "[НАЗВАНИЕ ФИЧИ]" 
для корпоративного хелпдеска.

Стек проекта:
- Frontend: React 18 + TypeScript + Vite + Tailwind CSS + shadcn/ui + Recharts + React Query (TanStack Query)
- Backend: Express + MySQL2 + JWT
- State: Zustand
- Routing: React Router v6
- Даты: date-fns

Требования к коду:
1. TypeScript — строгая типизация, никаких any
2. API-запросы через кастомный хук useApi с обработкой ошибок и тостами
3. Формы — React Hook Form + Zod для валидации
4. Загрузка данных — с skeleton-заглушками, а не "Loading..."
5. Оптимистичные обновления где уместно
6. Мобильная адаптивность обязательна

Формат ответа:
- Сначала краткий план реализации (3-5 шагов)
- Затем полный код файлов с путями относительно проекта
- SQL-миграции если нужны
- Не забудь про типы (interfaces/types)

Контекст текущей БД:
[вставь сюда актуальную схему из seed.sql]

Контекст текущего API:
[вставь сюда релевантные эндпоинты из server/routes/]
```

## 1. Дашборд — Статистика и графики

```
Реализуй дашборд хелпдеска.

Функционал:
- 4 карточки статистики: Всего тикетов, Открыто, В работе, Закрыто (с дельтой за неделю)
- Круговая диаграмма (Recharts PieChart) — распределение тикетов по статусам
- Линейный график — тикеты по дням за последние 7 дней
- Таблица "Последние обновления" — 5 последних тикетов с аватаркой автора

API эндпоинты:
GET /api/dashboard/stats — агрегированная статистика
GET /api/dashboard/activity — последние обновления

Требования:
- Карточки с анимацией чисел (count-up)
- Графики адаптивные, темная тема через Tailwind
- Данные кэшируются 60 секунд (React Query staleTime)
- Ошибки — fallback UI с кнопкой "Повторить"

Дай: компоненты, хуки для данных, типы, mock-данные для dev-режима.
```

## 2. Тикеты — CRUD + Фильтрация

```
Реализуй полный модуль тикетов.

Страница списка:
- Таблица с колонками: ID, Заголовок, Статус (бейдж), Приоритет (цвет), Исполнитель, Дата создания
- Пагинация (по 20 штук)
- Поиск по заголовку (debounce 300ms)
- Фильтры: по статусу (select), приоритету (select), исполнителю (select с поиском)
- Сортировка по клику на заголовок колонки
- Кнопка "Новый тикет" — открывает модалку

Модалка создания/редактирования:
- Поля: Заголовок*, Описание (textarea), Статус (select), Приоритет (select), Исполнитель (select из API /api/employees), Внутренние заметки (только для редактирования)
- Валидация Zod: заголовок min 5 символов, описание min 10

Страница детали тикета:
- Шапка с заголовком и метаданными
- Вкладки: Описание | История изменений | Внутренние заметки
- Блок комментариев (как мини-чат)

API:
GET /api/tickets?page=&search=&status=&priority=&assignee=&sort=
POST /api/tickets
GET /api/tickets/:id
PUT /api/tickets/:id
DELETE /api/tickets/:id
POST /api/tickets/:id/comments

Дай полную реализацию: страницы, компоненты, хуки, типы, валидацию, API-роуты на бэкенде.
```

## 3. Сотрудники — Список и онлайн-статус

```
Реализуй модуль "Сотрудники".

Функционал:
- Список с группировкой по отделам (аккордеон или секции)
- Карточка сотрудника: аватар, имя, должность, email, онлайн-статус (зеленая точка)
- Поиск по имени/email (фильтрует в реальном времени)
- Клик по сотруднику — сайдбар с деталями или переход на профиль

Онлайн-статус:
- Поле last_active в БД (timestamp)
- Считаем онлайн если last_active < 5 минут назад
- Обновляется при каждом запросе с JWT (middleware на бэкенде)

API:
GET /api/employees — список с группировкой по departments
GET /api/employees/:id — детали сотрудника
PUT /api/employees/:id/status — обновление статуса (для админа)

Требования:
- Аватарки — генерация через UI Avatars API если нет фото
- Группировка по отделам на фронте (reduce)
- Пустые состояния "Нет сотрудников в отделе"

Дай: компоненты, типы, SQL для departments, middleware обновления last_active.
```

## 4. Календарь — События

```
Реализуй календарь событий.

Функционал:
- Сетка месяца (7 колонок), навигация "← →" и выбор месяца/года
- Ячейка дня: число + точки событий (цвет по типу)
- Клик на день — модалка "Добавить событие"
- Клик на событие — модалка редактирования/удаления
- Боковая панель "Ближайшие события" (топ-5)

Модалка события:
- Название*, Дата*, Время, Тип (встреча/дедлайн/напоминание — цветная метка), Описание, Участники (multi-select сотрудников)
- Валидация: название min 3, дата не в прошлом

API:
GET /api/calendar?month=YYYY-MM — события месяца
POST /api/calendar/events
PUT /api/calendar/events/:id
DELETE /api/calendar/events/:id

Требования:
- date-fns для всех манипуляций с датами
- События кэшируются, инвалидация при CUD-операциях
- Drag-and-drop перенос события между днями (опционально, если сложно — skip)
- Мобильно: свайп месяцев

Дай: компонент CalendarGrid, хук useCalendarEvents, API-роуты, SQL таблицу events.
```

## 5. Опросы — Голосование

```
Реализуй модуль опросов.

Функционал:
- Список опросов: заголовок, статус (активен/завершен), дедлайн, кол-во голосов
- Карточка опроса: вопрос, варианты ответов (radio или checkbox), кнопка "Проголосовать"
- После голосования — показывать результаты: прогресс-бары с процентами
- Создание опроса (модалка): вопрос, тип (single/multiple), варианты (динамический список), дедлайн

Правила:
- Один пользователь = один голос (проверка по user_id)
- Если тип single — radio buttons, если multiple — checkboxes
- После дедлайна — только просмотр результатов

API:
GET /api/polls — список с флагом "я голосовал"
GET /api/polls/:id — детали + результаты если голосовал
POST /api/polls/:id/vote — { optionIds: number[] }
POST /api/polls — создание (админ)

Дай: компоненты PollCard, PollResults, хуки, валидацию, SQL со связью poll_votes.
```

## 6. Файлы — Хранилище

```
Реализуй файловое хранилище.

Функционал:
- Древовидная структура папок (breadcrumb навигация)
- Сетка/список переключение вида
- Загрузка файлов (drag-and-drop зона)
- Создание папки (модалка с именем)
- Фильтр по типу (все/изображения/документы/архивы)
- Поиск по имени файла
- Превью для изображений, иконки для типов файлов

Технические детали:
- Хранение: локально в /uploads с записью пути в БД
- Лимит: 10MB на файл
- Типы: image/*, .pdf, .doc, .docx, .xls, .xlsx, .zip

API:
GET /api/files?folderId=&search=&type=
POST /api/files/upload (multipart/form-data)
POST /api/files/folders
DELETE /api/files/:id
GET /api/files/:id/download

Дай: компонент FileManager, FileDropzone, хуки, multer-конфигурацию на бэкенде, SQL.
```

## 7. Чаты — Общие и личные

```
Реализуй модуль чатов.

Функционал:
- Список чатов слева: название, последнее сообщение, время, непрочитанные (badge)
- Окно переписки справа: сообщения с аватарками, реакции (эмодзи-пикер), удаление (только своих)
- Отправка: Enter, Shift+Enter для новой строки
- Поиск по сообщениям внутри чата
- Личные чаты: создаются при первом сообщении между двумя пользователями

Модель сообщения:
{ id, chatId, senderId, content, reactions: {emoji: userId[]}, createdAt, isDeleted }

API:
GET /api/chats — список чатов пользователя
GET /api/chats/:id/messages?beforeId=&limit=50
POST /api/chats/:id/messages
PUT /api/chats/:id/messages/:msgId/reaction
DELETE /api/chats/:id/messages/:msgId

Требования:
- Оптимистичная отправка сообщения (сразу в UI, откат при ошибке)
- Автоскролл к новому сообщению
- Поддержка markdown-ссылок в сообщениях
- Плейсхолдер "Напишите сообщение..." с аватаркой текущего пользователя

Дай: компоненты ChatList, ChatWindow, MessageBubble, хуки, типы, SQL.
```

- Переключатель темы (светлая/темная)

API:
GET /api/users/me
PUT /api/users/me
POST /api/users/me/avatar (multipart)
GET /api/users/me/files

Требования:

## 9. Авторизация — JWT + Dev-логин

```
Реализуй систему авторизации.

Функционал:
- Страница логина: Email, Пароль, "Запомнить меня", кнопка "Войти"
- Страница регистрации: Имя, Email, Пароль (min 8, 1 цифра, 1 заглавная), Подтверждение пароля
- Dev-логин: кнопка "Войти как разработчик" (только в dev-режиме, POST /api/auth/dev-login с userId=1)
- Защита роутов: приватные только для авторизованных, редирект на /login

JWT:
- Access token в httpOnly cookie или localStorage (выбери и обоснуй)
- Refresh token механизм (опционально)
- Интерсептор 401 → редирект на логин

API:
POST /api/auth/login
POST /api/auth/register
POST /api/auth/dev-login
POST /api/auth/logout
GET /api/auth/me

Требования:
- Хеширование паролей bcrypt (10 salt rounds)
- Защита от брутфорса: rate-limit 5 попыток за 15 минут
- Формы с валидацией Zod и отображением ошибок сервера

Дай: страницы Login/Register, AuthContext, защищенный HOC/компонент ProtectedRoute, middleware auth на бэкенде.
```
