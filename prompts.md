# Промты для разработки фич

> Используй как шаблоны при генерации/доработке кода через ИИ (Claude, GPT-4, Cursor и т.д.)

## Оценка и улучшение промтов

### Базовый шаблон (контекст)

**Оценка:** Хороший шаблон. Чёткий стек, конкретные требования, структурированный формат вывода. Но есть проблема — слишком длинный. DeepSeek/GPT-4 Flash могут «забыть» конец промта, особенно если контекст БД большой.

**Улучшение:** Разбей на 2 части. Сначала шаблон (без контекста БД), потом отдельным сообщением кидай схему и API.

### 1. Дашборд — Статистика и графики

**Оценка:** Чётко. Конкретные API, конкретные UI-элементы, конкретные требования. Модели такое любят.

**Что может сломаться:** «Карточки с анимацией чисел» — модель может навязать библиотеку `react-countup` вместо простого CSS-анимации. Уточни, если хочешь без лишних зависимостей.

### 2. Тикеты — CRUD + Фильтрация

**Оценка:** Самый сложный промт. Много функционала. Проблема: модель может сгенерировать всё в кучу — одним файлом на 500 строк. Или забудет про вкладки в деталях.

**Улучшение:** Разбей на 3 запроса:
1. Сначала список + фильтры + пагинация
2. Потом модалка создания/редактирования
3. Потом страница детали с комментариями

### 3. Сотрудники — Список и онлайн-статус

**Оценка:** Хорошо. «UI Avatars API» — конкретно, модель поймёт. Middleware для last_active — тоже чётко.

**Что может сломаться:** «Группировка на фронте (reduce)» — модель может сделать на бэкенде. Уточни, если важно.

### 4. Календарь — События

**Оценка:** «Drag-and-drop (опционально, skip)» — отлично. Даёшь модели выход. Иначе она может навязать `@dnd-kit` и сломать весь компонент.

### 5. Опросы — Голосование

**Оценка:** Правила чёткие. «Один пользователь = один голос» — модель поймёт, что нужен unique constraint в БД.

### 6. Файлы — Хранилище

**Оценка:** Хорошо. Multer — конкретно. Но модель может забыть про `multer.diskStorage` и написать `memoryStorage`, что сломает большие файлы.

### 7. Чаты — Общие и личные

**Оценка:** Сложный модуль. «Личные чаты создаются при первом сообщении» — модель может неправильно обработать логику создания чата. Может потребоваться уточнение.

**Что точно сломается без WebSocket:** реалтайм. Ты в TODO написал WebSocket — пока это polling или ничего.

### 8. Профиль — Редактирование

**Оценка:** «Автосохранение (debounce 2 сек) или кнопка» — модель выберет одно. Уточни, если важно. «Crop квадрат 200x200» — модель может навязать `react-image-crop`, хотя resize на бэкенде проще.

### 9. Авторизация — JWT + Dev-логин

**Оценка:** «httpOnly cookie или localStorage (выбери и обоснуй)» — модель может выбрать localStorage «проще», но это XSS-уязвимость. Лучше указать явно: httpOnly cookie.

### Почему DeepSeek 4 / Flash «фигово» справляются

| Проблема | Почему так | Решение |
|----------|-----------|---------|
| Забывают контекст | Промт 2000+ токенов, модель теряет начало | Разбивай на 2-3 сообщения. Сначала шаблон, потом схема БД, потом задача |
| Генерируют лишние зависимости | «Анимация чисел» → react-countup, «crop аватарки» → react-image-crop | Явно пиши «без новых библиотек, только CSS/нативные API» |
| Смешивают слои | Бэкенд и фронтенд в одном файле | Проси отдельно: «Сначала бэкенд (Express + SQL), потом фронтенд (React)» |
| Забывают edge cases | Нет обработки пустых состояний, ошибок сети | Добавь в промт: «Не забудь: empty state, error boundary, loading skeleton» |
| Типы кривые | any везде, несоответствие API и фронта | В шаблоне пропиши: «Типы из Types.ts синхронизировать с бэкендом» |

## Базовый промт-шаблон (для ЛЮБОЙ фичи)

```
Ты senior fullstack разработчик. Твоя задача — реализовать функционал "[НАЗВАНИЕ ФИЧИ]" 
для корпоративного хелпдеска.

Стек проекта:
- Frontend: React 18 + TypeScript + Vite + Tailwind CSS + shadcn/ui + Recharts + React Query (TanStack Query)
- Backend: Express + MySQL2 + JWT
- State: Zustand
- Routing: React Router v6
- Даты: date-fns

Требования к коду:
1. TypeScript — строгая типизация, никаких any
2. API-запросы через кастомный хук useApi с обработкой ошибок и тостами
3. Формы — React Hook Form + Zod для валидации
4. Загрузка данных — с skeleton-заглушками, а не "Loading..."
5. Оптимистичные обновления где уместно
6. Мобильная адаптивность обязательна

Формат ответа:
- Сначала краткий план реализации (3-5 шагов)
- Затем полный код файлов с путями относительно проекта
- SQL-миграции если нужны
- Не забудь про типы (interfaces/types)

Контекст текущей БД:
[вставь сюда актуальную схему из seed.sql]

Контекст текущего API:
[вставь сюда релевантные эндпоинты из server/routes/]
```

## 11. Уведомления — Real-time и история

```
Реализуй систему уведомлений для хелпдеска.

Функционал:
- Иконка колокольчика в шапке с бейджем (кол-во непрочитанных)
- Dropdown при клике: список последних 10 уведомлений, кнопка "Отметить все прочитанными"
- Типы уведомлений:
  • "Новый тикет назначен вам" → ссылка на тикет
  • "Новое сообщение в чате" → ссылка на чат
  • "Дедлайн события" → ссылка на календарь
  • "Системное" → без ссылки

Триггеры (автоматическое создание):
- При назначении тикета → уведомление исполнителю
- При новом сообщении в чате → всем участникам кроме отправителя
- За 1 час до события календаря → уведомление участникам

Техническая реализация (выбери один вариант):
**Вариант A (Polling):** GET /api/notifications каждые 30 секунд
**Вариант B (SSE):** Server-Sent Events для push-уведомлений
**Вариант C (WebSocket):** полный real-time (если в проекте уже есть WS для чатов)

Страница "Все уведомления":
- Таблица: тип, текст, дата, статус (прочитано/нет), действие (перейти/удалить)
- Фильтр по типу и статусу
- Кнопка "Очистить все"

API:
GET /api/notifications — список (с флагом unreadOnly)
PUT /api/notifications/:id/read
PUT /api/notifications/read-all
DELETE /api/notifications/:id
DELETE /api/notifications/clear-all
POST /api/notifications (внутренний, вызывается из других сервисов)

Требования:
- Уведомления хранятся 30 дней (cron-задача или TTL)
- На фронте: toast при получении нового уведомления (sonner)
- Мобильно: колокольчик в нижней навигации

Дай: компонент NotificationBell, страницу NotificationsPage, хуки useNotifications, API-роуты, SQL таблицу notifications, функцию createNotification для вызова из других модулей.
```

## 12. WebSocket — Real-time для чатов и уведомлений

```
Реализуй WebSocket сервер для real-time функционала хелпдеска.

Стек: Socket.IO (server + client)

Функционал:
- При подключении: аутентификация через JWT (передача токена в handshake)
- Комнаты: 
  • user:{userId} — личные уведомления
  • chat:{chatId} — сообщения в конкретном чате
  • ticket:{ticketId} — обновления тикета

События сервера → клиент:
- notification:new — новое уведомление
- chat:message — новое сообщение в чате
- chat:reaction — реакция на сообщение
- ticket:updated — изменение статуса/исполнителя тикета
- user:status — изменение онлайн-статуса сотрудника

События клиент → сервер:
- chat:join — вход в комнату чата
- chat:leave — выход из комнаты
- chat:typing — "печатает..." индикатор
- message:read — прочтение сообщения

Требования:
- При отключении пользователя — обновлять last_active и рассылать статус offline
- Оффлайн сообщения: если пользователь не в сети — уведомления накапливаются и отправляются при подключении
- Rate limiting: не более 50 сообщений в минуту на пользователя

Дай: серверный код (Socket.IO + Express integration), клиентский хук useSocket, контекст SocketProvider, интеграцию с существующими модулями чатов и уведомлений.
```
