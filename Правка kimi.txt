Критичные проблемы
1. handleResponse в api.ts — двойной res.json()
TypeScript
async function handleResponse(res: Response): Promise<any> {
  // ...
  const data = res.ok ? null : await res.json().catch(() => null)  // ❌ если !res.ok — читаем тело
  if (!res.ok) {
    throw new Error(data?.message || `Ошибка ${res.status}`)
  }
  // ...
  const payload = await res.json().catch(() => null)  // ❌ читаем тело ВТОРОЙ раз
Проблема: Response можно прочитать только один раз. Если res.ok === true, первый res.json() не вызывается (ternary), но если res.ok === false — тело уже прочитано, а потом пытаемся прочитать снова — будет ошибка.
Фикс:
TypeScript
async function handleResponse(res: Response): Promise<any> {
  if (res.status === 401 || res.status === 403) {
    localStorage.removeItem('token')
    localStorage.removeItem('user')
    window.location.href = '/login'
    throw new Error('Сессия истекла')
  }
  if (res.status === 204) return null
  
  const payload = await res.json().catch(() => null)
  if (!res.ok) {
    throw new Error(payload?.message || `Ошибка ${res.status}`)
  }
  if (payload && typeof payload === 'object' && typeof payload.success === 'boolean') {
    if (Object.prototype.hasOwnProperty.call(payload, 'data')) return payload.data
    return payload
  }
  return payload
}
2. tickets.js — getTicketById без проверки прав на чтение сообщений
В ticket-context.tsx mapTicket мапит messages из ответа API, но в tickets.js роут GET /:id возвращает messages без пагинации. Для тикетов с 1000+ сообщений — огромная нагрузка.
Фикс: разделить эндпоинты:
GET /:id — метаданные тикета (без messages)
GET /:id/messages — уже есть, с пагинацией
Или добавить ?includeMessages=false по умолчанию.
3. SocketContext — stale closure на socket
TypeScript
const sendMessage = useCallback((chatId: number, text: string) => {
  socket?.emit('message:send', { chatId, text })  // socket может быть null
}, [socket])
Если socket пересоздался между рендерами — callback устарел. Но тут [socket] в deps, так что ок. Настоящая проблема: socket state обновляется, но если компонент использует sendMessage в event listener без cleanup — stale closure.

5. TicketProvider — authFetch не обрабатывает 401
TypeScript
async function authFetch(url: string, token: string | null, options?: RequestInit) {
  const res = await fetch(url, { ...options, headers: { ...options?.headers, Authorization: `Bearer ${token}` } })
  if (!res.ok) throw new Error(`API error: ${res.status}`)
  return res.json()
}
При 401 не редиректит на логин, просто падает с ошибкой. api.ts это делает, но TicketProvider использует свой authFetch.
6. usePush — race condition на subscribe
TypeScript
const subscribe = useCallback(async () => {
  // ...
  setSubscribed(true)  // внутри try
  // ...
  setSubscribed(false)  // в catch
}, [supported])
Если пользователь быстро кликнет 2 раза — два параллельных запроса. Нужен AbortController или isSubscribing ref.
7. Kanban.tsx — DnD без dataTransfer fallback
TypeScript

12. server/src/socket.js — prisma.employees → prisma.employees?
JavaScript
prisma.employees.update({ where: { id: socket.userId }, data: { online: true } })
В Prisma schema модель называется Employee (PascalCase), но в клиенте — prisma.employee (camelCase)? Проверьте — в schema.prisma model Employee, значит в Prisma Client будет prisma.employee (единственное число, camelCase). Тут prisma.employees — опечатка, будет undefined.
13. background.js — warnAdminRedisMissing вызывается только при старте
Если Redis упал во время работы — админы не узнают. Нужен healthcheck в цикле.
14. files.js — requireRole('admin', 'senior_agent') на DELETE
В AGENTS.md написано "role check на DELETE (senior_agent+)", а в коде requireRole('admin', 'senior_agent'). requireRole использует Math.max по индексам иерархии, так что senior_agent пройдёт. Ок, но лучше явно requireRole('senior_agent').
