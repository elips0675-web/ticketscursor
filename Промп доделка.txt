Промпт: Доделка tickets-модуля (Code Review Fixes)
Контекст: Существующий проект Service Desk (Helpdesk) на React 18 + TypeScript + Vite + Express + Prisma + MySQL. Модуль тикетов частично переведён на сервисный слой (tickets.service.js), но роуты (tickets.js) всё ещё содержат бизнес-логику и имеют архитектурные проблемы.
Задача: Исправить все критичные проблемы, найденные в code review. Работать поэтапно, каждый пункт — отдельный коммит.
Этап 1. Исправить формат API-ответов (Критично)
Проблема: В роутах GET /:id, POST /, POST /:id/messages используется spread-оператор ...ticket, ...msg, ...mapped, что дублирует данные в ответе:
JavaScript
res.json({ success: true, data: mapped, ...mapped }) // ← мусор
Что сделать:
Убрать ...mapped, ...ticket, ...msg из всех res.json() в tickets.js
Единый формат ответа: { success: boolean, data: T } или { success: boolean, message: string }
Проверить, что фронтенд не ломается (он должен читать response.data.data, а не response.data.id)
Если фронт использует поля с верхнего уровня — исправить фронтенд-компоненты тикетов
Этап 2. Обработать ошибки уведомлений (Критично)
Проблема: notifyTicketCreated, notifyTicketAssigned, notifyStatusChanged, notifyPriorityChanged, notifyTicketMessage вызываются без await — ошибки теряются, тикет может создаться, но уведомление не отправится без логов.
Что сделать:
Обёрнуть все вызовы notify* в try/catch с logger.warn
Уведомления НЕ должны ломать основную операцию (создание/обновление тикета)
Добавить await перед уведомлениями
JavaScript
try {
  await notifyTicketCreated(ticket.id, req.user.name)
} catch (err) {
  logger.warn('notifyTicketCreated failed:', err.message)
}
Этап 3. Вынести бизнес-логику из роутов в сервисный слой (Критично)
Проблема: Роуты PUT /:id/priority, PUT /:id/assign содержат прямые запросы Prisma.
Что сделать:
Создать в tickets.service.js:
updateTicketPriority(id, priority) — возвращает { oldPriority, newPriority }
updateTicketAssignee(id, employeeId) — возвращает { oldAssignee, newAssignee, employeeName }
Перенести логику findUnique + update + logAudit + invalidateCache в сервис (или оставить logAudit/invalidateCache в роуте, но Prisma-запросы — только в сервисе)
Роуты должны вызывать сервис-функции и эмитить WebSocket-события

Этап 8. Исправить getResolvedAt для reopened (Средне)
Проблема: При статусе reopened поле resolved_at остаётся старым.
Что сделать:
JavaScript
function getResolvedAt(status, currentResolvedAt) {
  if (status === 'resolved' || status === 'closed') return new Date()
  if (status === 'reopened') return null
  return currentResolvedAt
}
