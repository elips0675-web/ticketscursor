Промт для AI-агента: Полный функциональный аудит
plain
Проведи комплексный функциональный аудит корпоративной системы Service Desk (helpdesk/ticketing system). Проект: React 19 + TypeScript + Vite + Tailwind v4 + shadcn/ui на фронтенде, Express 4 + Prisma 5 + MySQL 8 + Socket.IO 4 + Redis на бэкенде.

Проверь ВЕСЬ функционал систематически по следующей структуре:

═══════════════════════════════════════════════════════════════
1. АУТЕНТИФИКАЦИЯ И АВТОРИЗАЦИЯ (RBAC)
═══════════════════════════════════════════════════════════════

□ Логин по email/password
  - Валидация полей (пустые, невалидный email, короткий пароль)
  - Неверные credentials → 401
  - Успешный логин → JWT access token (15min) + refresh httpOnly cookie (7d)
  - Проверка: access token в localStorage, refresh в cookie

□ Регистрация
  - Валидация: email уникальность, пароль сложность, обязательные поля
  - Дублирующий email → 409 Conflict
  - После регистрации → автоматический логин или редирект на логин

□ Refresh token flow
  - Истёкший access → автоматический refresh через /auth/refresh
  - Истёкший refresh → редирект на логин
  - Refresh token rotation (если реализовано) — старый инвалидируется

□ Logout
  - Очистка localStorage + cookies
  - Инвалидация refresh token на сервере (чёрный список или удаление из БД)

□ LDAP/AD аутентификация (если настроено)
  - Bind + search
  - Auto-provisioning нового пользователя
  - Fallback на обычный логин при недоступности LDAP

□ Роли и права доступа (5 ролей: super_admin > admin > senior_agent > agent > requester)
  Для КАЖДОЙ роли проверь:
  - Какие страницы/разделы видны в sidebar/навигации
  - Какие API endpoints доступны (403 для недоступных)
  - Какие действия доступны (создание, редактирование, удаление)
  - Иерархия: super_admin проходит ВСЕ проверки, requester только свои тикеты

□ Защита роутов на фронтенде
  - ProtectedRoute редиректит неавторизованных на /login
  - Прямой доступ к /admin при роли agent → 403 или редирект

□ Защита API на бэкенде
  - Без токена → 401 на всех защищённых endpoints
  - Токен с недостаточной ролью → 403
  - Проверка: нельзя подделать роль в токене (подпись JWT)

═══════════════════════════════════════════════════════════════
2. ТИКЕТЫ (CORE ФУНКЦИОНАЛ)
═══════════════════════════════════════════════════════════════

□ Создание тикета
  - Обязательные поля: title, description, priority, category
  - Валидация: пустой title, слишком длинный title (>255), XSS в description
  - Автоназначение (AUTO_ASSIGN): выбирается наименее загруженный agent/senior_agent
  - SLA deadline автоматически рассчитывается: base × priorityMult × categoryMult
  - При создании → уведомление senior_agent+ и назначенному

□ Список тикетов
  - Пагинация (page, limit)
  - Фильтры: по статусу, приоритету, категории, assignee, дате
  - Сортировка: по дате создания, приоритету, дедлайну
  - Поиск: FULLTEXT / Meilisearch / LIKE fallback
  - requester видит ТОЛЬКО свои тикеты (проверь на бэкенде!)
  - agent видит свои + неназначенные
  - senior_agent+ видит все

□ Детальная страница тикета
  - Все поля отображаются корректно
  - История статусов (timeline)
  - SLA таймер (сколько осталось / просрочен)
  - Чат внутри тикета (внутренние заметки + публичные сообщения)
  - Вложения: preview, скачивание, удаление
  - Пагинация сообщений (take/skip)

□ Изменение статуса
  - open → in_progress → resolved → closed (валидные переходы)
  - resolved → reopened (сброс resolved_at)
  - Невалидные переходы → 400 (например, closed → open напрямую)
  - При изменении → уведомление участникам тикета

□ Изменение приоритета
  - low → medium → high → critical
  - При повышении приоритета → уведомление assignee + senior_agent+
  - Пересчёт SLA deadline при изменении приоритета?

□ Назначение / переназначение
  - Назначение на agent/senior_agent
  - Автоназначение при создании
  - Ручное назначение admin/senior_agent+
  - Уведомление новому assignee

□ SLA-мониторинг
  - first_response_at: фиксируется при первом переходе в in_progress
  - resolved_at: фиксируется при resolved/closed
  - dueAt: рассчитывается при создании, пересчитывается?
  - Просроченные тикеты: отображаются в админ-дашборде
  - Фоновая проверка каждые 15 мин (BullMQ или setInterval)
  - Эскалация: уведомление assignee + senior_agent+ (дедупликация 24ч)

□ Экспорт
  - CSV: UTF-8 BOM, все поля, корректные даты
  - PDF: jsPDF + html2canvas, форматирование, пагинация при большом тикете

□ Удаление
  - Мягкое или жёсткое удаление?
  - Каскадное удаление сообщений, вложений
  - Кто может удалять: admin+?

12. БЕЗОПАСНОСТЬ (ПЕНЕТРАЦИОННОЕ ТЕСТИРОВАНИЕ)
═══════════════════════════════════════════════════════════════

□ XSS
  - <script>alert(1)</script> в title/description тикета → экранируется?
  - HTML в wiki → sanitized?
  - Markdown поддержка? → XSS через атрибуты?

□ SQL Injection
  - Prisma защищает, но проверь $queryRaw с параметрами
  - Поиск: user input в FULLTEXT/LIKE

□ CSRF
  - POST/PUT/DELETE защищены JWT (stateless), CSRF неактуален для API
  - Но: формы login/register — CSRF токен?

□ IDOR (Insecure Direct Object Reference)
  - requester может получить чужой тикет по ID? (GET /api/tickets/123)
  - agent может получить тикет другого agent?
  - Проверь ВСЕ endpoints с :id параметром

□ Rate Limiting обход
  - Быстрые запросы с разных IP? (проверь реализацию)
  - DDoS на WebSocket?

□ File upload
  - .php, .jsp, .html → rejected?
  - Double extension: file.jpg.php → rejected?
  - Null byte: file.exe%00.jpg → rejected?
  - Magic bytes подмена?

□ JWT
  - Подпись проверяется (нельзя подделать payload)
  - Алгоритм none → rejected
  - Expired token → 401, не 500

ФОРМАТ ОТЧЁТА
═══════════════════════════════════════════════════════════════

Для КАЖДОЙ проверки укажи:
- ✅ PASS — работает корректно
- ❌ FAIL — есть баг или несоответствие требованиям
- ⚠️ WARN — работает, но есть риски или улучшения
- ⏭️ SKIP — не реализовано или нет доступа

Для FAIL и WARN:
1. Описание проблемы
2. Шаги воспроизведения
3. Ожидаемое vs фактическое поведение
4. Рекомендации по исправлению
5. Severity: Critical / High / Medium / Low

Итоговая оценка:
- Количество PASS / FAIL / WARN / SKIP
- Общий процент покрытия функционала
- Топ-5 критичных проблем
- Рекомендации по приоритизации исправлений
💡 Советы по использованию
Если AI-агент имеет доступ к коду:
bash
# Добавь в промт:
"Проверь код в директориях: src/, server/src/, e2e/. 
Для каждого модуля найди соответствующие файлы и проанализируй реализацию."
Если AI-агент может запускать тесты:
bash
# Добавь:
"Запусти: npm run test, cd server && npm run test, npx playwright test.
Проанализируй результаты и покрытие (npm run test:coverage)."
Если AI-агент может делать HTTP-запросы:
bash
# Добавь:
"Сделай тестовые запросы к API:
1. POST /api/auth/login с неверными credentials → ожидаю 401
2. POST /api/auth/login с валидными → ожидаю 200 + set-cookie
3. GET /api/tickets без токена → ожидаю 401
..."
Этот промт покрывает ~200+ проверок и даст исчерпывающий аудит всей системы.
