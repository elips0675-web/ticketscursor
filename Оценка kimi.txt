Вот детальный code review tickets.js и tickets.service.js.
tickets.js — Code Review
🔴 Критичные проблемы
1. Неконсистентный формат API-ответов
JavaScript
// Строка 59: дублирование данных
res.json({
  success: true,
  data: mapped,
  ...mapped,  // ← data и поля тикета на одном уровне — мусор в ответе
})

// Строка 81: аналогично
res.status(201).json({ success: true, data: ticket, ...ticket })

// Строка 178: тоже
res.status(201).json({ success: true, data: msg, ...msg })
Проблема: ...ticket/...msg/...mapped разворачивает все поля объекта в корень ответа. Получается { success, data, id, title, description, ... } — клиент получает данные дважды (в data и на верхнем уровне). Это баг, не фича.
Фикс:
JavaScript
res.json({ success: true, data: mapped })  // убрать spread
2. Отсутствие await при асинхронных операциях без обработки
JavaScript
// Строка 75: notifyTicketCreated — fire-and-forget без await
notifyTicketCreated(ticket.id, req.user.name)
if (autoAssignedTo) {
  notifyTicketAssigned(ticket.id, autoAssignedTo, req.user.name)
}
Проблема: Если уведомление упадёт (SMTP недоступен, Telegram API лагает) — ошибка не поймается, но и не обработается. Лучше await с try/catch или .catch() для логирования.
Фикс:
JavaScript
try {
  await notifyTicketCreated(ticket.id, req.user.name)
  if (autoAssignedTo) {
    await notifyTicketAssigned(ticket.id, autoAssignedTo, req.user.name)
  }
} catch (notifyErr) {
  logger.warn('Notification failed:', notifyErr)
  // не ломаем создание тикета из-за уведомления
}
3. getIO()?.emit() — silent fail
JavaScript
// Везде: getIO()?.emit('ticket:created', ...)
Проблема: Если Socket.io не инициализирован (getIO() вернёт null/undefined), эмит просто пропустится без лога. Клиент не узнает об обновлении.
Фикс:
JavaScript
const io = getIO()
if (io) {
  io.emit('ticket:created', { ...ticket, messages: [] })
} else {
  logger.warn('Socket.io not available, ticket:created not emitted')
}
4. Нет проверки прав на чтение тикета
JavaScript
// Строка 51: router.get('/:id', ...)
// Нет проверки: может ли текущий пользователь видеть ЭТОТ тикет?
Проблема: Любой аутентифицированный пользователь может прочитать любой тикет по ID. Нет проверки:
Является ли пользователь создателем?
Назначен ли он на тикет?
Есть ли у него роль admin/senior_agent?
Фикс: Добавить middleware или проверку в getTicketById:
JavaScript
// В сервисе или роуте:
if (req.user.role === 'agent' && ticket.assigned_to !== req.user.userId && ticket.created_by !== req.user.userId) {
  return res.status(403).json({ success: false, message: 'Forbidden' })
}

11. limit до 10 000 — потенциальная DDoS-уязвимость
JavaScript

12. Math.random() в имени файла — не криптографически безопасен
JavaScript

Проблема: Коллизии возможны при высокой нагрузке. Лучше crypto.randomUUID() или crypto.randomBytes(16).toString('hex').

1. getLeastLoadedAssignee — raw SQL без параметризации
JavaScript
// Строка 35-46: prisma.$queryRaw`...`
Проблема: Хотя здесь нет пользовательского ввода, prisma.$queryRaw без параметров — риск. Если когда-то добавите фильтр — SQL-инъекция.

  include: { _count: { select: { assigned_tickets: { where: { status: { in: ['open', 'in_progress'] } } } } } },

  take: 1,

Главные проблемы:
Дублирование данных в API-ответах (...ticket, ...msg, ...mapped) — баг
Fire-and-forget уведомления — без await и обработки ошибок
Бизнес-логика в роутах (updatePriority, updateAssign) — нарушение архитектуры
Нет проверки прав на чтение тикета — любой видит любой
Raw SQL без параметризации — риск SQL-инъекции при расширении
Загрузка всех сообщений в списке тикетов — потенциальная DDoS-уязвимость
Что хорошо:
Есть сервисный слой (начат)
Prisma используется корректно (типобезопасность)
Логирование присутствует (не везде, но есть)
Кэш-инвалидация на мутациях
Audit log на ключевых операциях
